# Validation Playbook

## 目的

验证的目标不是证明创作者的想法正确，而是用最小成本发现：谁有问题、问题有多急、当前怎么解决、愿意为什么结果付费、创作者能否交付。

## 验证对象

一次只验证一个陈述：

```text
我相信 [某类用户]
在 [具体场景]
反复遇到 [具体问题]
愿意为 [可验收结果]
接受 [交付方式和价格假设]
```

## 证据强度

由弱到强：

```text
点赞 / 收藏
< 评论说感兴趣
< 主动私信咨询
< 接受访谈
< 提供真实材料
< 支付定金
< 完整付款
< 完成任务
< 获得结果
< 再次购买 / 推荐
```

不要用弱证据替代强证据。

## 需求访谈

### 招募对象

- 最近 30 天确实遇到该问题；
- 已经尝试解决；
- 与目标用户画像匹配；
- 不只找朋友和忠实粉丝。

### 访谈问题

1. 最近一次遇到这个问题是什么时候？
2. 当时发生了什么？
3. 你已经尝试过什么？
4. 花了多少时间、金钱或机会成本？
5. 哪一步最卡？
6. 如果解决，什么变化对你最有价值？
7. 你如何判断它真的解决了？
8. 目前为什么还没有解决？
9. 你曾经为类似问题付费吗？
10. 如果有一个明确方案，你最担心什么？

不要在访谈前半段介绍产品，不要问“你会不会买我的课”。

## 最小 Offer 测试

一个可测试 Offer 必须包含：

- 谁；
- 什么问题；
- 什么结果；
- 多长时间；
- 如何交付；
- 需要用户投入什么；
- 真实价格；
- 名额或交付容量；
- 不适合谁。

### 测试方法

优先顺序：

1. 对已表达问题的用户一对一邀请；
2. 面向现有受众发布招募内容；
3. 召开一场免费说明会，但在现场提出真实付费 Offer；
4. 使用小额定金验证承诺；
5. 不用纯问卷替代销售。

## 拒绝原因分类

| 类型 | 常见表达 | 应对决策 |
| --- | --- | --- |
| 问题不急 | 以后再说 | 换更强场景，不急着降价 |
| 结果不清 | 到底能得到什么 | 重写承诺和验收方式 |
| 信任不足 | 你做过吗 | 补案例、过程和边界 |
| 交付不合适 | 没时间直播 | 调整节奏或异步比例 |
| 价格不匹配 | 太贵 / 太便宜不可信 | 检查人群、范围与支持成本 |
| 用户不匹配 | 我其实不需要 | 修正目标人群，不硬卖 |

## 14 天快速验证

### Day 1—2：定义假设

- 选择一个人群和问题；
- 写出结果与验收；
- 列出 20 位潜在访谈者。

### Day 3—5：访谈

- 完成至少 5 次访谈；
- 提取重复原话；
- 判断问题是否真实和紧急。

### Day 6—7：Offer

- 完成产品单页；
- 设计最小交付；
- 确定价格假设和容量。

### Day 8—10：销售测试

- 向匹配用户发出 10—20 次邀请；
- 记录打开、回复、通话、付款和拒绝原因；
- 不在两次拒绝后立刻改产品。

### Day 11—14：判断

结果分三类：

- `GO`：出现真实付款，问题与结果表达清楚；
- `ITERATE`：问题真实，但人群、结果、交付或价格有明确阻塞；
- `STOP`：问题不急、用户没有行动或创作者无法交付。

## 成功门槛

不要固定使用统一成交率。根据样本量和触达质量判断，但至少要求：

- 有真实的目标用户对话；
- 有真实价格；
- 有真实付款或明确拒绝；
- 有一次真实交付；
- 有结果或失败证据；
- 有下一轮清晰修改。

## 隐私与授权

- 默认不公开用户姓名、联系方式和商业数据；
- 公开案例前获得明确授权；
- 区分“允许内部复盘”和“允许公开引用”；
- 用户作品涉及公司或客户时，先完成脱敏；
- 不把测试用户的私人材料用于其他用途。
