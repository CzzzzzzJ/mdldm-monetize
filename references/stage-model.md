# Stage Model

## 目的

阶段模型用于阻止创作者过早录课、搭站或做会员。判断时选择“已有充分证据支持的最高阶段”，而不是用户希望达到的阶段。

## 阶段

### S0 Expression

信号：

- 内容主题经常变化；
- 没有稳定表达对象；
- 缺少具体成果；
- 用户互动无法归纳成问题。

唯一任务：找到一个自己确实做过、可以持续讲的问题。

### S1 Signal

信号：

- 已经持续发布；
- 有互动或私信；
- 但用户问题零散；
- 不确定谁最需要帮助。

唯一任务：收集 10—20 条真实用户问题，确认重复信号。

### S2 Validation

信号：

- 有重复问题；
- 有个人或他人成果；
- 用户对解决方案有兴趣；
- 但还没有稳定真实付款。

唯一任务：通过服务或小班完成第一次收费验证。

### S3 Delivery

信号：

- 已经有人付款；
- 用户获得了一定结果；
- 交付依赖创作者本人；
- 流程、任务和验收尚不稳定。

唯一任务：完整记录一次交付，识别重复动作和阻塞点。

### S4 Productization

信号：

- 已交付两轮或更多；
- 用户问题和关键动作重复；
- 已出现模板、清单、课程或 Skill 的雏形；
- 仍需降低重复讲解。

唯一任务：把重复交付变成标准产品。

### S5 Systemization

信号：

- 获客来源相对稳定；
- 产品和交付结构稳定；
- 用户需要持续更新、答疑或权益；
- 手工工具开始明显限制运营。

唯一任务：建立会员、知识站、成果和运营系统。

## 判断原则

- 粉丝多不等于阶段高。
- 一次爆款不等于需求成立。
- 免费咨询不等于付费验证。
- 单次成交不等于产品化。
- 搭了网站不等于系统化。

## 阶段转换 Gate

| 从 | 到 | 必须出现的证据 |
| --- | --- | --- |
| S0 | S1 | 连续内容 + 明确受众反应 + 可归类的问题 |
| S1 | S2 | 重复问题 + 能力证明 + 可描述的目标用户 |
| S2 | S3 | 真实付款 + 明确承诺 + 开始真实交付 |
| S3 | S4 | 至少两轮交付 + 重复任务 + 可观察结果 |
| S4 | S5 | 稳定获客 + 稳定产品 + 持续服务需求 + 系统成本合理 |

不能跳过 Gate。用户已经建站、建群或录课，也不能据此跳到 S5。

## 诊断置信度

### 高

- 有用户材料；
- 有付款或交付记录；
- 有明确结果；
- 关键信息互相一致。

### 中

- 有内容和部分用户信号；
- 缺少付款或交付细节；
- 可以给出带假设的阶段判断。

### 低

- 只有用户自述；
- 没有代表内容、用户问题或案例；
- 只能输出预诊断和下一步取证。

## 横跨阶段时如何处理

一个创作者可能在不同产品线上处于不同阶段。例如：

- 咨询已经 S4，但录播课仍是 S2；
- 内容账号已经 S1，但企业培训已经 S3；
- 会员已有用户，但持续服务仍未通过 S4 Gate。

此时不要给创作者一个笼统阶段。按“产品线”分别判断，再选择未来 30 天唯一主线。

## 阶段输出格式

```text
主线阶段：Sx
判断对象：具体产品 / 具体问题
置信度：高 / 中 / 低
已通过 Gate：
未通过 Gate：
当前唯一任务：
暂缓事项：
```
