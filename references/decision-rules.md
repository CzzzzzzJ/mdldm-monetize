# Decision Rules

## 从经验到规则

### 规则 1：课程多，但用户不知道学什么

不要继续增加课程。先设计用户身份、目标、学习路径、本周任务和结果。

### 规则 2：用户反复问同一个问题

先做一次付费诊断或小班，不要立即录制完整课程。

### 规则 3：用户听懂了，但没有行动

问题不在内容量，而在任务、反馈和验收。增加操作清单、作业和答疑。

### 规则 4：交付很累，每次都重复讲

记录重复环节，先模板化和 SOP 化，再考虑课程或 Skill。

### 规则 5：会员续费弱

检查用户是否有持续问题、固定更新和可感知权益。没有持续价值时，不要靠群维持会员。

### 规则 6：想搭知识站

先确认至少有两轮付费交付、重复材料、固定任务、答疑和成果。知识站是放大器，不是验证器。

### 规则 7：内容数据好，但没人买

检查内容解决的是“想看”还是“愿意付费解决”的问题。补充需求访谈和付费测试。

### 规则 8：粉丝少

不要自动判断无法变现。如果问题明确、能力可信、可以触达 5—10 个目标用户，优先服务型验证。

### 规则 9：粉丝多

不要自动建议做大课。先确认用户问题、支付意愿和交付结果。

### 规则 10：用户想一次做完所有产品

要求选择一个目标人群、一个问题、一个结果和一种交付方式，完成最小闭环后再扩展。

### 规则 11：用户问题高度个性化

先做 Service。不要强行把定制问题装进统一课程。

### 规则 12：同一问题需要练习和反馈

优先 Cohort。课程材料只承担输入，任务和反馈承担结果。

### 规则 13：同一问题已经被重复解决

可以进入 Product，但必须验证用户能否脱离作者独立完成。

### 规则 14：用户完成一次产品后仍持续回来

先识别持续问题和固定价值，再考虑 Membership，不要只建群。

### 规则 15：内容很专业，但普通用户听不懂

不要简单降低深度。先区分基础用户和进阶用户，分别设计入口与结果。

### 规则 16：小白和进阶用户同时存在

建立分层产品或路径，不要让一门课同时承担扫盲和完整项目。

### 规则 17：用户没有时间

不要默认缩短内容。减少同步环节、拆小任务、保留核心结果标准。

### 规则 18：用户完成率低

先检查入口匹配、任务大小、反馈机制和技术摩擦，再判断内容质量。

### 规则 19：用户想要商业化

不要先做抽象商业课。选择一个可发布、可服务或可出售的真实项目作为载体。

### 规则 20：用户成果很多但没有传播

建立成果授权、脱敏、案例结构和复盘流程，不要只收一句好评。

### 规则 21：创作者害怕首期价格低影响长期

把首期定义为验证价格，明确容量、范围和后续调整条件。

### 规则 22：首发没人买

不要立即降价。先区分触达不足、人群不匹配、问题不急、结果不清、信任不足和交付不合适。

### 规则 23：只有熟人购买

视为弱付费证据。下一轮必须触达非熟人或弱关系用户，测试真实需求。

### 规则 24：用户要求更多赠品

先确认核心结果是否清楚。赠品只能降低执行摩擦，不能代替核心价值。

### 规则 25：创作者想自动化获客

只有已经跑通的内容和转化流程才适合自动化。未验证前先人工观察。

### 规则 26：创作者有多个平台

选择一个主验证渠道，其他平台承担分发，不要为每个平台设计不同产品。

### 规则 27：已经有稳定企业客户

按企业服务产品线单独判断阶段，不用个人粉丝规模限制路线。

### 规则 28：产品依赖快速变化的平台或模型

课程负责稳定原理，会员或更新服务负责变化信息，避免整套课程频繁重录。

### 规则 29：用户只想要模板

确认模板能否独立产生结果。若必须理解场景和做判断，增加诊断或使用边界。

### 规则 30：用户要求保证结果

拒绝收入或商业结果保证。可以承诺交付动作、反馈次数和验收标准。
